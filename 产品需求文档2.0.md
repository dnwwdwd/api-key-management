# 产品需求文档 (PRD)：Api-Key管理 (v2.0 资产洞察版)

## 1. 产品升级概述 (V2.0 Vision)

**升级背景：** 在静态安全存储的基础之上，V2.0 旨在为开发者提供“API 资产大盘”。通过主动查询与本地数据分析，让用户对分散在各个大模型厂商的 API Key 的状态、额度和消耗情况一目了然。同时优化了底层数据架构，将 `Base URL` 与厂商强绑定，提升管理效率。

**核心理念：** 坚持完全本地化的管理工具定位，不做 API 代理，所有数据分析均在本地浏览器/本地 Next.js 服务端完成。

**全局体验：** 深度集成国际化 (i18n)，全站提供无缝的**简体中文 (zh-CN)** 与 **英文 (en-US)** 双语切换体验，打造世界级的开发者工具。

## 2. 核心新增与优化功能 (V2.0 Features)

### 2.1 架构优化：Base URL 厂商级绑定与初始化

- **核心变更：** `Base URL` 属性从 `api_keys` 表转移到 `providers` (厂商) 表中。同一个厂商下的所有 API Key 将共享该厂商配置的 Base URL。新建 API Key 时不再需要填写该字段。
- **内置初始化：** 系统首次启动时，内置的主流 AI 厂商（如 OpenAI, DeepSeek, 月之暗面, 智谱等）的官方 `Base URL` 将被**默认初始化填充好**，用户开箱即用。
- **选填机制：** 新增或编辑自定义厂商时，`Base URL` 设为选填项，给予用户最大的配置自由度。

### 2.2 厂商管理与侧边栏操作面板 (Provider Management)

- **交互入口：** 侧边栏的 AI 提供商列表中，悬浮（Hover）在具体厂商上或点击“更多”按钮时，展示快捷操作面板。
- **操作功能：**
  - **修改 (Edit)：** 允许直接修改该提供商的名称和 `Base URL`。
  - **删除 (Delete) 与安全防呆限制：** 为了防止数据孤岛，**仅当该厂商下关联的 API Key 数量为 0 时，才允许执行删除操作**。若该厂商下仍有绑定的 API Key，删除按钮须被置灰禁用，或在点击时弹出拦截提示：“*请先删除或转移该厂商下的所有 API Key！*”

### 2.3 密钥一键测活与延迟检测 (API Connectivity Test)

- **功能描述：** 在 Key 详情页增加「网络测活 / Ping Test」按钮。只测试网络连通性，不提供聊天 UI。
- **强制前置校验：** 点击测活前，系统**必须校验其所属厂商是否已填写 `Base URL`**。若未配置，需阻断测试并弹出提示弹窗引导用户去填写：“💡 *测试失败：当前厂商未配置 Base URL，请先在侧边栏编辑该厂商信息并补充请求地址。*”
- **执行逻辑（强制后端代理）：** 前端调用自身的 Next.js 后端 API（如 `/api/test-key`）。由**后端服务器**携带 API Key 向该厂商的 `Base URL + /v1/chat/completions` (或标准测试路径) 发送极简非聊天探测包（如 `max_tokens: 1`），严禁前端直接跨域请求 AI 厂商。
- **结果展示与异常处理：**
  - 🟢 **成功：** 展示 HTTP 200 及延迟毫秒数。
  - 🔴 **失败：** **必须优先透传并展示 AI 厂商接口返回的原始错误信息**（通常为英文，如：`"error": {"message": "Incorrect API key provided..."}`），随后在 UI 下方追加**跟随当前语言环境 (Locale)** 的友好排障提示。

### 2.4 密钥额度与用量精确查询 (Balance & Usage Check)

- **UI 动态展示与防呆策略 (Graceful Degradation)：**

  - **白名单机制（系统内置）：** 系统需在底层代码中固定维护一个支持查询的厂商白名单（该名单为系统固定，不允许用户手动填写或维护）。仅当选中的 API Key 属于白名单时，详情页才展示「查询余额/用量」按钮及数据卡片。
  - **不支持时的多语言提示：** 对于不支持查询的厂商，隐藏查询按钮，并显示多语言友好的文案提示：“ℹ️ *该厂商采用企业级云计费或暂未提供公开 API，不支持直接通过 API Key 查询额度。*”
  - **查询异常处理：** 如果查询接口请求失败（如 401/404），**优先展示厂商 API 返回的原始错误详情**，随后补充多语言排障提示。

- **支持的厂商、API 规范及执行要求：**

  - **强制要求：** 所有查询余额和用量的请求，必须通过 Next.js 后端 Route Handlers 发送（如 `/api/check-balance`），严禁前端直接跨域请求。

  1. **DeepSeek (深度求索)**: `GET https://api.deepseek.com/user/balance` (余额)
  2. **Moonshot (月之暗面 / Kimi)**: `GET https://api.moonshot.cn/v1/users/me/balance` (余额)
  3. **OpenAI (及部分兼容中转站)**: `GET https://api.openai.com/v1/dashboard/billing/usage` (用量)
  4. **智谱 AI (Zhipu)**: `GET https://open.bigmodel.cn/api/paas/v4/tools/account/tokens` (用量)

### 2.5 独立数据看板页面 (Analytics Dashboard)

- **功能描述：** 新增独立的“统计洞察 / Analytics”路由页面，用于全局统揽所有 API 资产。所有图表标签、表头必须全面支持 i18n。
- **数据刷新策略（实时获取）：** **每次用户进入或刷新该看板页面时，系统将并发调用**所有白名单内 API Key 的余额/用量查询接口，确保大盘数据为实时最新状态。在数据并发拉取期间，页面需展示优雅的骨架屏 (Skeleton) 或全局加载状态，避免出现白屏。
- **UI 布局要求 (强依赖 shadcn/ui)：**
  - **顶部 - 核心指标卡 (Metric Cards)**
  - **中部 - 可视化图表 (Visual Charts)：** 强制使用 shadcn/ui 的 Charts 组件 (底层为 Recharts)。
  - **底部 - 资产明细表格 (Data Table)：** 对于不支持查询额度的厂商，其用量和余额列统一显示为 `-` 。

### 2.6 可用模型清单拉取 (Model Discovery)

- **功能描述：** 提供拉取当前厂商及 Key 可用模型权限列表的能力，作为脚手架生成的底层依赖。
- **执行逻辑（强制后端代理）：** 所有模型数据的拉取，必须由前端请求自身后端接口（如 `/api/fetch-models`），由**后端服务**发起对 AI 厂商的真实探测请求，以彻底规避前端跨域问题。
- **接口通用性与适配规范：** 目前大多数厂商为了降低接入成本，采用了与 OpenAI 完全一致的 API 规范。
  - **标准兼容阵营 (OpenAI, DeepSeek, Moonshot, 智谱V4, 零一万物等)：** * **请求方法：** `GET`
    - **URL：** 厂商配置的 `Base URL` + `/models` (或 `/v1/models`)
    - **解析逻辑：** 提取返回 JSON 中的 `data` 数组里的 `id` 字段。
  - **特殊厂商阵营 (Google Gemini)：**
    - **URL：** `GET https://generativelanguage.googleapis.com/v1beta/models`
  - **静态兜底策略 (Fallback - 针对不支持的厂商如 Anthropic Claude)：** 对于官方未提供动态模型拉取接口的厂商，系统应在前端代码中**硬编码**维护一份常用模型白名单清单（如 `claude-3-5-sonnet-latest`, `claude-3-opus-latest` 等），当识别到该类厂商时，直接返回静态清单，而不发起真实网络请求。

### 2.7 多语言开发脚手架 (Snippet Generator - 弹窗版)

- **功能描述：** 为开发者免去翻阅官方文档的繁琐，直接提供本地生成的 API 调用代码片段。
- **UI 展示机制：** **必须使用 shadcn/ui 的 Dialog (弹窗) 组件**进行展现。严禁采用单独路由页面或将长篇代码块直接平铺在 Key 详情页的 div 中，保持详情页的极简清爽。
- **执行逻辑与交互工作流：**
  - **入口：** 在 Key 详情页点击「获取调用代码 / Code Snippets」按钮，唤出脚手架弹窗。
  - **前置数据拉取与选择：** * 弹窗渲染前或渲染时，系统调用 2.6 章节的模型拉取能力。
    - 在弹窗内部提供一个基于 shadcn/ui 的**支持搜索的下拉框 (Combobox)** 供用户选择拉取到的 AI 模型。
    - **选填特性：** 模型选择允许为空（用户可跳过此步）。若不选，系统则在生成的代码模型字段处插入提示性占位符（如 `your-model-name-here`）或使用官方默认模型（如 `gpt-4o`）。
  - **代码生成：** 用户通过弹窗内的另一个下拉框/Tabs 选择编程语言（如 `cURL`, `Python`, `Node.js`）。系统结合 **Base URL**、下拉框选中的 **模型名称 (或占位符)** 以及当前详情页的 **明文 API Key**，动态拼接代码。
- **结果展示：** 弹窗内实时显示生成的排版精美、带高亮的只读代码块。下方提供一键「复制到剪切板」的按钮，点击后 Toast 提示成功。

## 3. 国际化 (i18n) 专项规范

为确保产品的全球化开发者体验，V2.0 必须严格遵循以下国际化标准：

- **技术方案：** 采用 `next-intl` 结合 Next.js App Router 方案。
- **语言支持：** 默认语言为简体中文 (`zh-CN`)，全面支持英文 (`en-US`)。提供全局的语言切换按钮。
- **数据与格式本地化 (Localization)：** 不对 AI 厂商返回的原始报错 (Message) 强行机翻，保留原汁原味；仅对应用自身生成的提示与 UI 文字进行双语适配。

## 4. 技术与架构设计要求

### 4.1 数据库结构更新 (Drizzle ORM Schema)

- **`providers` (厂商表):**
  - `id`
  - `name` (厂商名称)
  - `base_url` (选填字段，系统初始化时为主流厂商提供默认值)
  - `is_custom` (是否为用户自定义)
- **`api_keys` (密钥表):**
  - `id`
  - `provider_id` (FK，外键关联 `providers`)
  - `name`
  - `api_key` (加密存储)
  - `notes`
  - *(注：移除旧版的 base_url 字段，调用时通过联表关联获取)*

### 4.2 性能与安全边界 (强制后端代理)

- **彻底规避 CORS 与防泄露机制：** 绝对禁止前端直接向第三方 AI 厂商（如 `api.openai.com` 等）发起 HTTP 请求。
- **统一收口要求：** **所有的第三方外网交互（包括连通性测试、拉取可用模型、查询额度/用量），必须统一使用 Next.js 的 Route Handlers 作为本地反向代理**（例如创建 `/api/check-balance`, `/api/test-key`, `/api/fetch-models` 接口）。前端只需向应用自身的这些同源 API 发起请求，由后端 Node.js 环境负责组装鉴权 Header 并发出真实的外部请求，将结果返回给前端。
- **响应式适配：** 独立的数据看板页面需确保在中大屏幕上左右并排图表，在移动端变为上下堆叠。
