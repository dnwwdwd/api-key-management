# 产品需求文档 (PRD)：Api-Key 管理

## 1. 文档说明

本文档用于描述当前版本的产品目标、功能范围、交互规则和技术边界。内容以仓库现有实现为基础，同时作为后续迭代的统一参照。

## 2. 产品定位

Api-Key 管理是一款本地优先的 API Key 管理工具，主要服务于需要同时使用多家 AI 服务的开发者。

产品目标：

- 安全保存并管理多家 Provider 的 API Key。
- 让录入、复制、查看、测试和排查操作尽量集中在一个界面完成。
- 在不暴露密钥的前提下，提供测活、余额或用量查询、模型列表获取和调用代码生成能力。

本产品不是业务流量网关，也不承担用户正式请求的代理转发职责。相关接口仅用于管理、测试和辅助开发。

## 3. 目标用户与使用场景

目标用户：

- 个人开发者
- 需要同时维护多家模型服务账号的技术团队
- 希望统一保存 API Key、Base URL 和备注信息的使用者

典型场景：

- 管理多个 Provider 和多个 API Key
- 快速复制密钥并查看关联的 Base URL
- 判断某个 Key 是否可用
- 查询部分 Provider 的余额或用量
- 获取当前 Provider 支持的模型列表并生成示例代码

## 4. 功能范围

### 4.1 登录与会话

- 系统使用账号密码登录。
- 仅使用 Credentials Provider。
- 登录态基于 Auth.js 管理。
- Session 有效期固定为 48 小时。
- 未登录用户访问受保护页面时，必须跳转到 `/{locale}/login`。

### 4.2 国际化

- 支持 `zh-CN` 与 `en-US`。
- 登录后可在账号菜单中切换语言。
- 用户当前语言偏好会记录到用户表中的 `preferred_locale` 字段。
- UI 文案需要支持两种语言；第三方服务返回的原始错误信息保持原样展示，不强制翻译。

### 4.3 Provider 管理

- 系统首次进入主页面时，会补齐内置 Provider 列表。
- 预置 Provider 包含常见 AI 服务和若干兼容 OpenAI 风格接口的平台。
- Provider 字段包括：
  - `name`
  - `base_url`
  - `is_custom`
- 用户可以新增自定义 Provider。
- 用户可以编辑 Provider 名称与 `base_url`。
- 用户可以删除 Provider，但前提是该 Provider 下没有任何 API Key。
- Provider 展示为纯文本，不使用图标字段。

### 4.4 API Key 管理

- 一个 Provider 可以关联多个 API Key。
- API Key 字段包括：
  - `name`
  - `provider_id`
  - `api_key`
  - `notes`
  - `created_at`
  - `updated_at`
- `api_key` 入库前必须加密，数据库中不保存明文。
- 页面支持新增、编辑、删除 API Key。
- API Key 列表按最新记录优先展示。
- 列表项中展示名称、Provider、Base URL、API Key、备注和更新时间。
- API Key 默认明文展示，并支持显隐切换。
- 用户可一键复制 API Key。

### 4.5 连通性测试

- 用户可对单个 API Key 发起测活请求。
- 测活前如果 Provider 没有配置 `base_url`，应直接拦截。
- 前端只能调用站内接口 `/api/test-key`。
- 后端使用当前 API Key 向目标 Provider 发送最小化请求，并返回状态码和延迟。
- 若失败，优先返回上游服务的原始错误信息，再由前端补充本地提示。

### 4.6 余额与用量查询

- 查询入口通过 `/api/check-balance` 提供。
- 并非所有 Provider 都支持查询，系统需要按 Provider 类型判断是否显示或执行该能力。
- 当前实现中：
  - `DeepSeek`、`Moonshot/Kimi` 返回余额类信息
  - `OpenAI` 返回用量类信息
- 对不支持的 Provider，前端需要明确提示而不是给出错误印象。
- 对不支持 API 查询的 Provider，应提供官方账单或用量页面的跳转链接，便于用户直接查看。

### 4.7 模型列表与代码片段

- 用户可在代码片段弹窗中请求模型列表。
- 模型发现通过 `/api/fetch-models` 由服务端执行。
- 普通 OpenAI 兼容 Provider 优先尝试 `/models` 或 `/v1/models`。
- Gemini 走单独的模型查询地址。
- Anthropic 优先调用官方 `/v1/models` 接口。
- 当厂商模型接口不可用、鉴权失败或返回空结果时，系统需要回退到内置的厂商推荐模型列表。
- 用户可选择语言生成调用示例，当前支持：
  - cURL
  - Python
  - Node.js
- 代码片段由前端根据 `base_url`、模型名和 API Key 直接拼接，并支持复制。

### 4.8 数据分析页面

- 系统提供独立分析页：`/{locale}/analytics`。
- 分析页进入后并发请求可查询的 Key 数据。
- 页面显示内容包括：
  - Key 总数
  - 可查询 Key 数
  - 查询成功数
  - 聚合后的数值总量
  - 柱状图
  - 环形图
- 数据拉取过程中展示骨架或占位状态。

### 4.9 设置

- 设置入口统一放在账号菜单中。
- 设置弹窗包含两个主要区域：
  - 修改密码
  - Provider 管理
- 修改密码时需要校验旧密码。
- 修改成功后需要重新登录。

## 5. 界面与交互要求

### 5.1 视觉风格

- 整体使用黑、白、灰配色。
- 错误提示可使用红色，成功提示可使用绿色。
- 主按钮保持深色背景与浅色文字。
- 页面以轻边框和留白为主，不依赖厚重阴影。

### 5.2 桌面端布局

- 桌面端采用三栏结构：
  - 左侧：应用导航与账号菜单
  - 中间：Provider 列表
  - 右侧：API Key 单列表内容区
- 左侧导航至少包含：
  - API Key 页面
  - Analytics 页面
- 账号菜单触发器需要展示当前登录账号。

### 5.3 移动端布局

- 移动端采用单栏布局。
- 主要操作入口需要保持可点击面积足够。
- 不使用依赖鼠标悬浮才可见的关键操作。

### 5.4 列表与缺省值规则

- API Key 页面使用单列表组件，不拆成“列表 + 详情”双区结构。
- 每条记录应展示完整信息和完整操作入口。
- Provider 的编辑和删除入口收纳在“更多”菜单中。
- 任意无值字段统一显示 `--`。
- 时间字段只显示时间值，不额外显示字段标签文案。
- 测活成功后，延迟小于 `500ms` 显示绿色，大于等于 `500ms` 显示红色。

### 5.5 复制与反馈

- 复制行为使用 `navigator.clipboard`。
- 成功或失败都需要通过轻量 Toast 给出反馈。

## 6. 数据模型

### 6.1 users

- `id`
- `username`
- `password_hash`
- `preferred_locale`
- `created_at`

### 6.2 providers

- `id`
- `name`
- `base_url`
- `is_custom`

### 6.3 api_keys

- `id`
- `provider_id`
- `name`
- `api_key`
- `notes`
- `created_at`
- `updated_at`

补充说明：

- 当前版本中，`base_url` 存在于 `providers` 表，不存在于 `api_keys` 表。
- `api_key` 存储的是密文，前端拿到的是服务端解密后的值。

## 7. 安全与技术边界

- 不允许前端直接请求第三方 AI 服务。
- 所有外部交互必须经过 Next.js Route Handlers。
- 当前核心后端接口包括：
  - `/api/test-key`
  - `/api/check-balance`
  - `/api/fetch-models`
- 加密算法使用 Node.js `crypto` 的 AES-256-GCM。
- 加密密钥从环境变量 `ENCRYPTION_KEY` 读取。
- 数据库存储与本地运行方式应保持“本地优先”原则。

环境变量约定：

- `AUTH_SECRET`
- `ENCRYPTION_KEY`
- `DATABASE_FILE`
- `SEED_ADMIN_USERNAME`
- `SEED_ADMIN_PASSWORD`

补充说明：

- 本地开发允许使用默认开发密钥快速启动。
- 正式部署时必须显式配置认证密钥与加密密钥。
- 项目根目录提供 `.env.example` 作为本地配置模板。
- 仓库默认将 `data/app.db` 纳入版本管理，用于保证多机拉取同一提交时，初始化数据库内容保持一致。

## 8. 非功能要求

- 保持 TypeScript 严格类型约束。
- 主要页面在桌面端和移动端都应可用。
- 页面文案、表单校验、错误提示需要支持中英文。
- 文档变更要与实现保持同步。
